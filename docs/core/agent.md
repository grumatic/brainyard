# Agent Component — Overview

`components/agent` is Brainyard's full-lifecycle runtime for autonomous AI agents:
goal-directed reasoning, controlled tool-use, durable memory, multi-agent
coordination, human-in-the-loop (HITL) safety, and first-class observability.

Top namespace: `ai.brainyard.agent`
Public API: `components/agent/src/ai/brainyard/agent/interface.clj`

The component is organized around a single substrate — a **behavior tree (BT)** —
whose leaves call DSPy signatures, registered tools, sandbox evaluations, or
human prompts. Every reasoning style (ReAct, CoAct, and the specialist agents
built on top of them) is just a different subtree.

## The Agent record

`core/agent.clj` defines a small `defrecord` (`[agent-id !state !session]`)
implementing five protocols from `core/protocol.clj` plus `java.io.Closeable`:

- **IAgent** — `agent-id`, `agent-name`, `agent-description`, `user-id`,
  `session-id`, `defagent-type`, `process`, `get-tools`, `get-state`.
- **IAgentLifecycle** — `start-agent`, `stop-agent`, `agent-running?`,
  `clone-agent` (shared session, snapshotted state, optional BT override).
- **IAgentState** — access to the BT, BT context, and `st-memory-init` atom.
- **IAgentBTIntegration** — callbacks the BT extended nodes invoke via Java
  interop: `update-session-data`, `check-run-cancelled?`,
  `check-run-paused?`, `await-resume`, `create-action-promise`,
  `get/set-action-permission`, `get-bt-st-memory`.
- **IAgentMemoryAccess** — `get-memory-manager`. (Recall/remember are plain
  fns in `agent/core/memory.clj`, not protocol methods.)
- **java.io.Closeable** — orderly teardown.

Two atoms back the record:

- **`!state`** — per-agent: runtime config, BT, memory manager, Clojure-agent ref,
  `st-memory-init`, analytics flags.
- **`!session`** — *shared with sub-agents* — the agent-session:
  `:user-id`, `:session-id`, `:messages`, `:data` (traces, todo-info, artifacts,
  exceptions, user-action), `:agent-activity` ring-buffered stream, `:turn-id`,
  session config.

### Two-layer session model

- **Agent session** (`!session`) — conversational identity shared between a
  root agent and its sub-agent family. Carried through `setup-agent`'s
  `:agent-session {:user-id :session-id}` kwarg. Always present; inherited
  by sub-agents from their parent.
- **App session** — the UI/bridge layer (TUI tab, WebSocket connection).
  Tracks permission-fn, dirs, feedback-fn, scrollback, watches. A single
  app session can host multiple concurrent agent-sessions. The agent
  component never sees the app-session-id — it is a concern of
  `bases/agent-tui` and `bases/agent-web`.

### Agent registry

`core/agent.clj` maintains a flat registry: `{agent-id agent}`. Agent ids
are globally-unique **namespaced keywords** of the form
`:<defagent-type>/<suffix>` (e.g. `:coact-agent/crimson-parrot-42`),
generated by `generate-instance-id`. `setup-agent-by-id` auto-generates one
when the caller omits `:id`. Because the defagent-type lives in the
namespace portion, `(proto/defagent-type agent)` recovers it without a
secondary index.

Accessors:

- `(list-agents)` — all live instances.
- `(list-agents-for-session session-id)` — filter by agent-session.
- `(get-agent agent-id)` — direct lookup.
- `(reset-agent-registry!)` — clear (testing).

Three dynamic vars (`*current-agent*`, `*call-depth*`, `*call-chain*`)
carry agent identity across thread-local calls and enforce multi-agent
safety (see [tool.md](tool.md)).

| `schedule-agent` | `common/schedule_agent.clj` | Front door for **time**-triggered prompt jobs (`schedule$*`) — plain-language intent → a validated cron/one-shot spec, previewed as plain words plus a concrete next-fire. Watches are excluded (event-agent owns them) |
| `event-agent` | `common/event_agent.clj` | Front door for the flat **event** vocabulary (`event$*` / `reaction$*` / `watch$*`) — declares events, wires `trigger → action` reactions, authors autonomous watch pollers, validates the wiring (payload schema, `{{key}}` vars, `:match` keys) before persisting, and diagnoses a dead rule |
| `state-machine-agent` | `common/state_machine_agent.clj` | Front door for user-defined **FSMs** — definition CRUD (`fsm$define`/`remove`/`list`) plus the per-session runtime (`fsm$send`/`status`/`reset`), two lifecycles kept distinct. `coact-agent` and `react-agent` are the reasoning primitives. The rest are
specialist BT configurations built from the same primitives
([reasoning.md](reasoning.md)) — most of the specialists run a CoAct-derived
tree (`coact/run-coact-derived`).

The last three — schedule-agent, event-agent, state-machine-agent — give the
event subsystem a conversational CRUD surface and introduce **zero new
commands**: each only orchestrates command families that already ship. The
boundaries are drawn so a request routes to exactly one (time-triggered jobs
vs. the flat event vocabulary vs. stateful FSMs), and they compose by calling
each other by name at the seams. All three *read* their gate flags
(`:enable-scheduler`, `:enable-reactions`, `:enable-fsm`, `:fsm-allow-code`)
via `agent-runtime$config` but never write them — a gate change is a config
write, handed to `config-agent`. Each also treats its markdown dossier under
`.brainyard/agents/<agent>/dossiers/` as a hard contract: a write-producing
turn that ends without one is an incomplete turn, enforced by a FINAL-STEP
checklist in the instruction. Designs:
[schedule](../design/schedule-agent-design.md) ·
[event](../design/event-agent-design.md) ·
[state-machine](../design/state-machine-agent-design.md).

## Hooks registry

`core/hooks.clj` is the single application-facing subscription point for
agent-runtime events. Apps (TUI, web bridge, plugins, analytics observers)
register handlers against the catalog; the runtime fires them at the
appropriate lifecycle points.

Event catalog (selected entries — see `event-catalog` in source for the
complete list):

| Event | Payload | Semantics |
|---|---|---|
| `:agent.session/created` / `/resumed` / `/closed` | `{:session-id :user-id :session}` | agent-session lifecycle |
| `:agent.instance/created` | `{:agent}` | after creation or clone, once started |
| `:agent.instance/closed` | `{:agent}` | before `.close` tears the agent down |
| `:agent.ask/pre` / `/post` | `{:agent :input (:result)}` | around `process` inside `ask` |
| `:agent.ask/finalize` | `{:agent :input :result}` | **gated** — fires after the result is built, before `/post`; a `:replace` can inject an auto-persist handoff line |
| `:agent.iteration/pre` / `/post` / `/exhausted` | `{:agent :iteration :max-iterations …}` | BT main-loop boundary |
| `:agent.dspy-action/pre` / `/chunk` / `/post` | `{:agent :node-id :signature …}` | each DSPy LLM call (streaming chunks via `/chunk`) |
| `:agent.tool-calls/pre` / `/post` | `{:agent :iteration :calls (:results)}` | a batch of tool calls |
| `:agent.tool-use/pre` | `{:agent :tool-name :args :call-id :depth}` | **gated** — handler decision can block/replace/modify |
| `:agent.tool-use/post` | `{:agent :tool-name :args :call-id :result}` | after the tool returns |
| `:agent.code-eval/pre` / `/post` | `{:agent :code :backend (:result :output :error :duration-ms)}` | CoAct code-block evaluation (`:backend` ∈ `:sandbox` / `:nrepl`) |
| `:agent.context/budgeted` | `{:agent :total-tokens :budget :section-tokens :compactions :over-budget?}` | context budget recomputed |
| `:agent.compaction/pre` / `/phase` / `/post` | `{:agent :before-tokens :after-tokens :trigger …}` | context compaction lifecycle (`/phase` ∈ `:previous-turns` / `:agent-context` / `:iterations`) |
| `:agent.evaluation/*` | `{:agent …}` | evaluation pipeline progress |
| `:agent.analytics/post` | `{:agent :analytics}` | post-session analytics computed |
| `:task/created` / `:task/completed` | `{:task}` | TaskManager lifecycle |
| `:todo/updated` | `{:agent :todo-list :active-slug}` | todo-agent's active list changed |
| `:agent/exception` | `{:agent :phase :exception}` | unhandled exception escaped a runtime phase |

### Gated vs. observer events

Most events are fire-and-forget. Two — `:agent.tool-use/pre` and
`:agent.ask/finalize` — are **gated**: handlers return a decision map and
the caller acts on the first non-`:allow` decision. (`:agent.ask/finalize`
is how rlm-agent's safety-net injects its `Saved …:` handoff line into the
finalized result before observers on `:agent.ask/post` see it.)

```
{:result :allow | :block | :replace | :modify-args
 :reason "..."                   ;; required when not :allow
 :replacement any                ;; when :replace
 :args        {...}              ;; when :modify-args
 :answer      "..."}             ;; optional, consumed by :block
```

The default loop guard (`common/loop_guard_hook.clj`) registers itself as
a `:agent.tool-use/pre` gate to short-circuit pathological tool loops.

### Streaming LLM chunks

Streaming token deltas are **not** a hook event. BT LLM-calling actions
invoke `agent.core.bt/chunk-factory-handler` directly with
`{:agent :st-memory-atom :node-id}` to obtain a per-node chunk handler
that writes deltas into st-memory under `:llm-streaming-text`. UIs (TUI,
web bridge) watch that key to render streaming output.
`behavior-tree/dspy-action` and `clj-sandbox` reach the factory via
`requiring-resolve` so they don't hard-depend on the agent component.

### API

```clojure
(register-hook!  event-key handler-id handler-fn
                 & {:keys [match priority source on-error]})
(unregister-hook! event-key handler-id)
(unregister-source! source-kw)        ; bulk teardown for a plugin
(list-hooks)  /  (list-hooks event-key)
(reset-hooks!)
(fire!          event-key event-map)         ; observer events — void
(fire-decision! event-key event-map)         ; gated events — returns decision
```

Match helpers: `match-agent-id`, `match-defagent-type`, `match-root-agent`,
plus `match-all` / `match-any` combinators. Exception policy is per-entry
`:on-error :log` (default) / `:throw` / custom fn.

## Multi-agent coordination

Sub-agents share the parent's `!session` and forward their display-stage
activity into the parent's `:agent-activity` stream with a monotonically
increasing `:seq` (diffable from the TUI). Two safeguards live in
`core/protocol.clj`:

- **`*call-depth*`** — capped by `:max-agent-call-depth` (default 3).
- **`*call-chain*`** — rejects calls where the target agent-id already
  appears in the chain, preventing cycles.

Messages persisted by sub-agents are tagged with `:agent-id`, preserving
attribution in transcript replays.

### Macro-scale orchestration

Two CoAct-derived defagents thread a durable dossier across multiple
specialist calls:

- **workflow-agent** (`common/workflow_agent.clj`) — domain-specific
  multi-stage workflows. Templates under `.brainyard/workflows/<id>.edn`
  declare stage sequence and workflow-level acceptance. Dossier under
  `.brainyard/agents/workflow-agent/<id>/`.
- **research-agent** (`common/research_agent.clj`) — multi-specialist
  research threads composing explore / plan / todo / exec / eval / update.
  Dossier under `.brainyard/agents/research-agent/<id>/`.

Both replace the original `pipeline/` DAG executor and `autoresearch/`
hill-climbing loop (retired 2026-05). The CoAct loop, the dossier, and a
curated tool roster make the LLM responsible for sequencing, retries, and
termination instead of a separate executor BT.

## Runtime, queueing, and cancellation

`core/runtime.clj` attaches a `clojure.core/agent` to each Agent record.
`ask-async` dispatches via `send-off`, serializing turns per agent while
permitting concurrent *different* agents. The executing thread ref is
captured in runtime state; `cancel-run` sets `:cancelled?` (polled by BT
nodes) and interrupts the thread, propagating up the parent chain.

**Action permissions** use promises: `create-action-promise` returns a
deref-able promise that BT nodes block on; the TUI/Web UI delivers the
user's response via `deliver-action`, with "always" / "never" answers
cached in `:action-permissions` for reuse.

`core/queue.clj` provides a cancellable work queue for agent invocations
where ordering and backpressure matter (notably the TUI).

## Human-in-the-loop

Two layers route through the same action-permission primitives, giving a
uniform surface for frontends:

- **In-BT HITL** — `request-user-action`, `user-approval-action`
  (Yes / No / Always-Yes / Always-No), `user-interrupt-action`
  (Continue / Stop). These are ordinary BT leaves that block on an
  action promise and update session thinking. See [bt.md](bt.md).
- **Conversational HITL** — workflow-agent collapses the historical
  5-mode HITL knob into one config (`:auto | :gates | :checkpoint |
  :co-pilot | :step`). Gates surface as ordinary `:answer`-channel
  pauses ending with `Awaiting workflow gate: <stage-id>`; the user
  replies, the loop resumes on the next turn.

## Configuration

`core/config.clj` defines a typed **runtime-config schema** covering loop
control, memory, safety, LM selection, analytics, and UI. Reads and writes
flow through a single API (`get-config` / `set-config!`) over a five-layer
precedence chain — schema defaults → global `!global-config`
(`.brainyard/config.edn`) → session config → per-agent override →
environment variable (a key's `:env-fn`, which wins over every persisted
layer). The same namespace resolves `.brainyard/` directory layout (plans,
worktrees, memory DB) with project vs. user scoping.

See [config.md](config.md) for the schema reference, state-slot layout,
persisted-EDN shape, and directory resolution details.

## Observability

Three layers, BT step → turn → session:

- **Trace** (`common/trace.clj`) — depth-indented BT traces threaded into
  session thinking for UI rendering.
- **Trajectory** (`common/trajectory.clj`) — structured per-iteration
  record (code, output, reasoning, cost, model, timing) suitable for
  fine-tuning corpora.
- **Turn log** (`common/log.clj`) — queries structured mulog events
  by user / session / agent / turn.
- **Evaluation** (`common/evaluation.clj`) — `EvaluateAnswer` and
  `FinalizeAnswer` DSPy signatures feeding the CoAct refinement loop and
  the ReAct finalize node.

The `analytics` component (referenced via `requiring-resolve`) computes
post-session PQS scoring, waste-pattern detection, and cost calculation.

## Design principles

1. **Reasoning is data.** Behavior trees + DSPy signatures make every
   thought inspectable.
2. **One registry, many flavours.** `deftool` / `defcommand` /
   `defskill` / `defagent` share storage, dispatch (`invoke-tool`), and
   conversion (`def->tool`).
3. **Code as the action space.** CoAct turns executable Clojure (and
   bash / python / js) into the orchestration language.
4. **Memory is layered and compactable.** Short-term inside the BT,
   long-term in FTS5, with compaction on both sides.
5. **Agents compose safely.** Shared sessions, depth + cycle caps,
   propagated cancellation.
6. **Humans are first-class.** HITL primitives in the BT plus
   conversational gates on the answer channel.
7. **Everything is observable.** Traces, trajectories, mulog logs, and
   scores from BT tick to session.
8. **Nothing is hard-wired.** Protocols, registries, `requiring-resolve`,
   typed runtime-config.
